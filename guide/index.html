    CLI-first documentation

    Cheri is now a collaboration CLI with a worker API behind it.

    The guide now centers on registration, login, workspace selection, file sync flows, task automation, team invites, activity, provider setup, and the security model. Browser approval pages, dashboard-first UX, and worker-rendered UI are out of scope.

    CLI command overview

    Register / login / logout

    Bootstrap identity, re-enter an existing session, and cleanly remove local state.

    cheri register

    Backend config

    Cheri resolves the Worker API URL from environment variables, saved local config, or deployment metadata derived from Wrangler files.

    cheri config get
cheri config set api-url https://cheri.parapanteri.com
cheri config reset
cheri config check

    Upload / download / list

    File movement is grant-based and defaults to the active workspace, with one-off --workspace overrides when needed.

    cheri file upload .\notes.txt
cheri file list
cheri file download notes.txt

    Teams and activity

    Invite collaborators, inspect membership, revoke invite generations, and review activity.

    cheri teams invite
cheri teams list
cheri teams invite-reset
cheri activity

    Task automation

    Create local sync tasks for files and directories. New tasks start background watching immediately unless you pass --no-start.

    cheri task create --directory cheri_test_files --mode on-change
cheri task start <task-id>
cheri task stop <task-id>
cheri task list
cheri task run <task-id>

    Workspace switching

    One authenticated user can keep multiple workspaces, list them, and explicitly change the saved active target.

    cheri workspace

    Automation

    Task automation

    Task commands

    cheri task create --directory cheri_test_files --mode on-change
cheri task create --directory "C:\Users\Name\Desktop\cheri_test_files" --mode on-change
cheri task find cheri_test_files
cheri task start <task-id>
cheri task stop <task-id>
cheri task remove <task-id>
cheri task logs <task-id>

    Runtime posture

    • Task definitions, runtime snapshots, and logs stay local to the operator machine.
    • New tasks auto-start a background watcher and can still be overridden with task watch.
    • Directory tasks preserve relative paths such as src/app.js in the workspace file registry.
    • Summary events are mirrored into workspace activity after each run.

    Defaults

    • File tasks default to instant.
    • Directory tasks default to on-change with debounce.
    • upload-only is the current safe default direction.
    • Recursive directory watching is enabled by default.

    Safety controls

    • Paths are normalized before task creation.
    • Simple names are resolved from the current folder, Desktop, Documents, and Downloads.
    • Temporary and editor-noise files are excluded by default.
    • Stopped tasks do not run until started again.
    • Errored tasks stop automatic execution until the operator intervenes.

    Identity

    Auth and bootstrap flow

    Register

    1. Ask for username.
    2. Ask for the first workspace name.
    3. Ask for a storage provider.
    4. Require confirmation for System (recommended).
    5. Generate a 12-word bootstrap secret.
    6. Create the session and active workspace.
    7. Offer local save or copy-only output.

    Login

    • Refreshes a saved valid session when possible.
    • Falls back to username + bootstrap secret when needed.
    • Uses cheri workspace join <invite-code> for the clean join flow.
    • Never depends on a browser approval page.
    Bootstrap secret scope

    The 12-word bootstrap secret is only for the user's own bootstrap and login identity flow. It is not reused as a team invite.

    State handling

    Local save vs copy behavior

    Save locally

    • Stores public state and sensitive credentials in separate files.
    • Keeps the session token out of the general workspace summary file.
    • Only stores the bootstrap secret when the operator explicitly chooses to save it.
    • Uses a storage abstraction so encrypted-at-rest support can be added later.

    Copy instead

    • Does not persist local state.
    • Prints a copy-friendly JSON payload.
    • Leaves secret handling explicit and operator-controlled.
    Explicit by design

    Cheri does not assume local persistence. The user must opt in to save credentials on the machine.

    Collaboration

    Invite code format

    Current format

    CHR-TEAM-8X2K91QZ

    Invite codes are short, workspace join tokens created by admins.

    Lifecycle

    • Stored as invite records in the backend.
    • Accepted via CLI, not an HTML invite page.
    • Revoked by cheri teams invite-reset.
    • Replacement codes can be minted immediately after reset.
    • Separate from session tokens and bootstrap secrets.
    No phrase reuse

    Invite codes never reuse the 12-word bootstrap identity format. That separation reduces operator confusion and keeps join credentials distinct from login credentials.

    Workspace model

    Workspace and multi-project behavior

    Active workspace

    cheri workspace
cheri workspace list
cheri workspace create --name team-alpha
cheri workspace use team-alpha
cheri workspace join CHR-TEAM-8X2K91QZ

    Without a subcommand, Cheri lists accessible workspaces. workspace create creates and selects. workspace use switches the saved active workspace. workspace join accepts a team invite and switches to the joined workspace.

    Multi-project access

    • One user can belong to multiple workspaces.
    • The session remains user-scoped, not workspace-scoped.
    • Invite acceptance adds membership instead of replacing prior access.
    • File and team commands default to the active workspace and support --workspace overrides without rewriting it.

    Security

    Security notes

    Bootstrap secret

    12 words. Used only for user bootstrap and login identity.

    Session token

    Bearer session credential for active CLI access.

    Invite code

    Short workspace join token for team onboarding.

    Provider credentials

    Workspace storage integration material, not user auth.

    What the current repo hardens

    • Short-lived upload and download grants.
    • Membership validation on file and team actions.
    • Admin checks for invite creation and invite reset.
    • Explicit workspace overrides do not silently change the saved active workspace.
    • Provider summaries redact secret fields.
    • Worker responses avoid exposing raw storage internals.

    Operator expectations

    • Do not log bootstrap secrets or session tokens.
    • Treat invite codes as join credentials.
    • Treat provider keys as workspace integration secrets.
    • Use local save only on trusted machines.

    Sync model

    Sync roadmap and versioning strategy

    What exists now

    • Workspace-scoped file registry.
    • Version increments on same-name uploads.
    • Uploader, editor, checksum, and timestamps in metadata.
    • Activity feed for uploads, modifications, membership, and invite events.
    • Local task automation with persisted definitions and run logs.

    Roadmap direction

    • Local-to-remote sync loops.
    • Conflict detection hooks.
    • Encrypted local state.
    • External provider connectors.
    • Richer session rotation semantics.
    Versioning strategy

    Cheri keeps a stable file id and appends version metadata when the same file name is uploaded again, preparing the backend for future conflict-aware sync instead of one-off artifact transfer.
